# 项目模块面试复习 Q&A

本文按简历模块整理。回答时建议先讲业务问题，再说明设计取舍，最后给出验证方式或量化结果；`[待补充]` 项在完成压测或故障演练后替换为真实数据。

## 一、知享知识分享平台

### 1. 登录与会话安全

**Q：为什么不用单个 JWT，而要做双令牌和 Redis 校验？**

A：纯 JWT 在签发后通常只能等过期，用户修改密码、退出登录或发现令牌泄露时无法立即作废。短期访问令牌用于接口鉴权，长期刷新令牌用于续期；刷新令牌在 Redis 中保留可控状态，因此可以按设备或全局撤销。代价是多一次 Redis 访问，但换来了可撤销性和更可控的会话管理。

**Q：RS256 的价值是什么？**

A：RS256 使用私钥签名、公钥验签。验签方只需持有公钥，避免多个服务共享签名密钥，适合后续服务拆分。小型单体系统使用对称签名也可行；这里选择 RS256 是为降低密钥扩散风险，而不是为了堆砌算法。

**Q：如何说明效果？**

A：补充登录、鉴权、刷新和撤销接口的 P95/P99 延迟，以及 Redis 可用性异常时的降级策略与测试结果。

### 2. 高并发互动

**Q：为什么点赞、收藏不直接写 MySQL？**

A：热点内容会让同一行计数和关系表承受高频写入，造成锁竞争和响应抖动。Redis 先承接实时状态与计数，异步聚合再落库，将用户请求路径与批量持久化路径分开。

**Q：位图和 Lua 分别解决什么？**

A：位图适合用一个 bit 表示“某用户是否已操作”，节省空间并支持快速判断；Lua 将“判重、更新状态、更新计数”放在 Redis 内原子执行，避免多条命令之间并发穿插而重复计数。

**Q：最终一致性会不会让用户看到错误数据？**

A：用户刚操作后的实时展示优先读 Redis，数据库的统计值允许短暂滞后。需要明确异步落库失败后的重试、对账和补偿策略，并用“写入削减比例、积压量、最终收敛时间”证明设计效果。

### 3. 关注事件同步

**Q：为什么使用 Outbox，而不在业务代码中直接发 Kafka？**

A：直接发消息会遇到“数据库提交成功但消息没发出”或“消息发出但事务回滚”的双写不一致。Outbox 与关注关系在同一个 MySQL 事务中落库，Canal 从 binlog 读取已提交的 Outbox 记录，再投递 Kafka，从而保证业务事实与待发送事件同时持久化。

**Q：消息重复投递怎么办？**

A：把事件 ID 作为消费幂等键；消费端记录已处理事件或使用幂等更新。消息系统通常提供至少一次投递，因此消费者必须按“可能重复”设计，而不是假设只会收到一次。

**Q：要监控哪些指标？**

A：Outbox 未投递数量、Kafka 消费积压、端到端延迟、失败重试次数和对账差异数。

### 4. 热点内容缓存治理

**Q：三级缓存为什么合理？**

A：本地缓存延迟最低，适合短时间内频繁读取的数据；Redis 让多实例共享缓存；MySQL 是最终数据源。层级的目标是把绝大多数热点请求挡在更靠近应用的一层，同时保留跨实例一致的缓存能力。

**Q：如何处理缓存击穿、穿透和雪崩？**

A：击穿用请求合并，让同一热点失效时只有一个请求回源；雪崩用随机过期时间分散失效点；穿透可用空值缓存或布隆过滤器。实现时要设置合理 TTL，避免把不存在的数据长期缓存。

**Q：为什么只说“提升稳定性”，不直接写性能翻倍？**

A：没有压测依据的倍数没有说服力。应补充缓存命中率、数据库 QPS、接口 P95/P99 和峰值并发前后的对比，作为简历中的真实指标。

### 5. 可追溯的智能问答

**Q：Resource / Session / Memory 三层分别是什么？**

A：Resource 是原始知识资料；Session 是一次问答过程中的即时上下文；Memory 是跨轮复用的已提炼信息。分层的原因是三类数据的生命周期、更新频率和检索需求不同，混在一起会让检索结果失焦。

**Q：分层 Markdown 索引解决了什么？**

A：文档标题、章节和具体段落粒度不同。先用粗粒度定位主题，再下钻到段落，可在召回范围与答案精度间平衡，也能将答案映射回原文位置。

**Q：如何验证 RAG 质量？**

A：准备人工标注的问题—证据集，评估检索命中率、答案引用正确率和无依据回答比例；不要只用模型主观评分。

## 二、RepoAgent

### 1. 可恢复的任务执行

**Q：为什么 Session 之外还要有 TaskState？**

A：Session 描述对话连续性，Task 描述一个具体目标的执行生命周期。一个会话可以包含多个任务；任务需要有状态、步骤、预算、等待审批和终止原因，不能只靠聊天记录推断。

**Q：追加式事件日志和任务快照如何配合？**

A：任务快照便于快速读取当前状态；事件日志按顺序保留发生过什么，便于审计和恢复。状态变化时将事件、快照和预算放在同一 SQLite 事务提交，避免只更新一半。

**Q：乐观锁为什么适合这里？**

A：用版本号比较更新前后的状态。当多个进程或恢复操作尝试修改同一任务，只有版本匹配的更新能成功，冲突方重新读取后决定重试或退出，避免静默覆盖。

### 2. 工具副作用控制

**Q：什么是“崩溃一致”的 ToolAction？**

A：执行有副作用工具前，先持久化动作、幂等键和执行阶段；执行完成后再记录结果。即使进程在中间崩溃，恢复流程也能判断动作尚未执行、已完成或结果不确定，而不是盲目再执行一次。

**Q：哪些错误可以自动重试？**

A：只读或明确幂等的瞬时失败可以有限重试，例如网络超时；可能写文件、发请求且不能确认结果的非幂等操作不自动重试，转为等待人工确认或标记“不确定副作用”。

**Q：为什么要把异常统一成 Observation？**

A：模型不应解析各种 Python 异常文本。统一字段如状态、错误类型、是否可重试、耗时和证据引用，能让策略层基于稳定语义决策，也便于追踪和统计。

### 3. 长任务上下文管理

**Q：什么是“近期工具证据回放”？**

A：指从历史工具调用中挑选和当前任务有关的结果摘要，重新加入本轮上下文，例如已读取的文件路径、测试失败原因和已执行命令。它不是把全部日志塞回模型，而是让模型在恢复后继续利用关键事实。

**Q：为什么要有 token 预算和滚动摘要？**

A：上下文窗口有限，完整历史会增加成本并稀释重点。token 预算控制每类信息占用空间；较早历史压缩为摘要，保留目标、结论、未决问题和关键证据，以便为最近的原始内容腾出空间。

**Q：如何评估上下文工程有效？**

A：比较相同任务下的 token 消耗、重复检索/重复工具调用次数、任务完成率和恢复后首次有效动作比例。

### 4. 可观测性与测试

**Q：Trace 记录哪些信息，为什么不是普通日志？**

A：Trace 以任务 ID 和关联 ID 串起模型调用、决策、审批、工具尝试、观察结果、预算与状态迁移。普通日志偏向文本排错；结构化 Trace 可以回答“为何执行这个动作、执行过几次、何时耗尽预算、失败在哪一步”。

**Q：硬预算的意义是什么？**

A：限制模型调用、工具调用、Token 和总时长，防止异常循环持续消耗资源。重复决策和连续失败也应触发停止或一次反思重规划，使失败可控、成本可预期。

**Q：222 项测试和 90.65% 覆盖率如何讲得有分量？**

A：说明覆盖的风险点：状态迁移、事件顺序、版本冲突、事务回滚、跨进程审批、重试边界、预算和 Trace 完整性。覆盖率是信号，不等于质量；要结合集成测试与故障注入说明关键路径真实被验证过。

## 三、开源贡献

**Q：OpenViking 的修复本质是什么问题？**

A：流式消息与角色信息可能乱序到达；若异常时过早清理会话状态，尚未确认提交的内容会丢失。修复将“清理”放在确认提交成功之后，并兼容乱序输入，保证消息持久化和记忆提取的完整性。

**Q：这段经历体现了什么能力？**

A：不是只改一个异常分支，而是识别了流式处理中的时序与提交确认边界，补齐失败路径，并通过上游 PR 审核和合并验证方案可被真实项目接受。

## 四、面试表达模板

**Q：如何在 40 秒内介绍一个模块？**

A：按“场景—问题—方案—结果”回答：

> 在【业务场景】下，原方案会因为【具体问题】导致【用户或系统影响】。我负责设计【核心方案】，关键取舍是【为什么选它】；最终【真实数据/待补充指标】，并通过【测试、监控或对账】验证。

**Q：没有压测数据时怎么回答？**

A：如实说明“当前已完成功能和自动化验证，性能指标正在按并发量、P95/P99、缓存命中率/消息延迟等维度补测”，再说明压测方案。不要把预期收益写成已达成结果。
